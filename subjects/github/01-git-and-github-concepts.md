# Git 与 GitHub 概念：本地与远程结构

> 搞清「Git」和「GitHub」分别是什么，以及「本地仓库」「远程仓库」「工作区 / 暂存区」在流程里各代表什么。

- **标签**：`#Git` `#GitHub` `#版本控制` `#概念`
- **日期**：2025-03-11
- **来源**：Ad-Studium 学习笔记

---

## 1. 核心概念 / 定义

| 术语 | 一句话定义 |
|------|------------|
| **Git** | 安装在你自己电脑上的**版本控制软件**，负责在本地记录每次修改（提交 commit）、分支、历史等。 |
| **GitHub** | 一个**网站/服务**，用来在互联网上存放「远程仓库」；你可以把本地的 Git 仓库推上去，也可以从上面克隆下来。 |
| **仓库（Repository）** | 一个被 Git 管理的项目文件夹，里面除了你的文件，还有隐藏的 `.git` 目录，存着所有版本历史。 |
| **本地仓库** | 在你电脑上的那个带 `.git` 的文件夹。 |
| **远程仓库** | 在 GitHub（或别的服务器）上的同一个项目的副本，通过 URL 和本地关联。 |

---

## 2. 要点 / 步骤

1. **Git** = 本地工具；**GitHub** = 远程托管平台；先有 Git，再和 GitHub 连起来。
2. 一次「上传」的完整路径：**本地改文件 → 提交到本地仓库 → 推送到远程（GitHub）**。
3. 本地 Git 里有三个重要概念：**工作区**、**暂存区**、**本地仓库（提交历史）**；理解这三块，后面的 `add` / `commit` 就不会懵。

---

## 3. 细节与推导

### 3.1 为什么需要「本地」和「远程」？

- **本地**：只有你能改、速度快、可离线工作。
- **远程**：备份、多人协作、在别的电脑上也能拿到最新代码。

所以流程是：在本地用 Git 做好「一次修改的打包」（commit），再用 `push` 把这次打包发到 GitHub；反过来，用 `pull` 把别人或别的设备上的更新拉回本地。

### 3.2 本地 Git 的「三层结构」

这是理解后续每一步的关键：

```
┌─────────────────────────────────────────────────────────┐
│  工作区（Working Directory）                              │
│  就是你看到的项目文件夹里的文件，直接编辑的就是这里。         │
└─────────────────────────────────────────────────────────┘
                            │
                            │  git add
                            ▼
┌─────────────────────────────────────────────────────────┐
│  暂存区（Staging Area / Index）                           │
│  「准备要放进下一次提交」的文件列表，相当于一个购物车。        │
└─────────────────────────────────────────────────────────┘
                            │
                            │  git commit
                            ▼
┌─────────────────────────────────────────────────────────┐
│  本地仓库（Local Repository）                             │
│  已经打好的「快照」（commit），存在 .git 里，有完整历史。     │
└─────────────────────────────────────────────────────────┘
                            │
                            │  git push
                            ▼
┌─────────────────────────────────────────────────────────┐
│  远程仓库（Remote，例如 GitHub）                          │
│  服务器上的仓库，大家都能访问。                             │
└─────────────────────────────────────────────────────────┘
```

- **工作区**：你改的只是「文件本身」，Git 还没把这些改动记下来。
- **暂存区**：用 `git add` 把「要纳入本次提交」的改动放进暂存区，相当于勾选「这些算一次更新」。
- **本地仓库**：用 `git commit` 把暂存区里的内容打成一条**提交记录**，永久进入历史。
- **远程**：用 `git push` 把本地仓库里新增的提交，上传到 GitHub。

所以：**先 add（选哪些改动能提交），再 commit（打成一条记录），最后 push（把记录传到 GitHub）**。后面几篇会按这个顺序展开。

---

## 4. 总结 / 与其它知识的联系

- **Git** 管的是你电脑上的版本历史；**GitHub** 是存「远程仓库」的地方，上传、下载、协作都围绕「本地 ↔ 远程」进行。
- 本地操作的核心是：**工作区 → 暂存区（add）→ 本地仓库（commit）**；上传则是 **本地仓库 → 远程（push）**。
- 下一篇会讲：在 GitHub 网页上「创建仓库」时，你其实是在创建那个「远程仓库」的空壳，然后再用本地去连它。
