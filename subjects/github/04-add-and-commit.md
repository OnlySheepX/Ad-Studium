# 工作区 → 暂存区 → 提交：add 与 commit

> 把「改动的文件」变成「一条可追溯的提交记录」要分两步：先 `git add`（放进暂存区），再 `git commit`（打成一次快照）。每一步在动哪一层、为什么这样设计，这里说清楚。

- **标签**：`#Git` `#add` `#commit` `#暂存区` `#工作区`
- **日期**：2025-03-11
- **来源**：Ad-Studium 学习笔记

---

## 1. 核心概念 / 定义

| 术语 | 含义 |
|------|------|
| **工作区** | 你直接编辑的目录里的文件；修改、新建、删除都发生在这里，Git 不会自动记录。 |
| **暂存区（Staging Area / Index）** | 一个「准备区」：你通过 `git add` 把要纳入**下一次提交**的改动放进来，commit 只针对暂存区里的内容。 |
| **git add** | 把工作区里指定文件（或全部）的**当前改动**标记为「要进入下一次提交」，写入暂存区。 |
| **git commit** | 把**当前暂存区**里的内容打成一条**永久提交记录**，写入本地仓库历史，并需要你写一条「提交说明」（message）。 |

---

## 2. 要点 / 步骤

1. 改完文件后，用 `git status` 看哪些被改了、哪些还没进暂存区。
2. 用 `git add` 把要提交的改动放进暂存区（可以按文件 add，也可以 `git add .` 全部）。
3. 用 `git commit -m "说明"` 把暂存区打成一条提交，并写清楚这次改了什么。
4. 此时改动已经进入**本地仓库**，但还没上传到 GitHub；上传要用 `git push`（见 05、06）。

---

## 3. 细节与推导

### 3.1 为什么要有「暂存区」？

- 一次修改里，你可能想「只提交一部分文件」或「只提交某文件的一部分」（通过 `git add -p`）。暂存区就是让你**挑选**「这次提交里要包含哪些改动」。
- 没有暂存区的话，就只能「全部改了什么一起提交」，无法精细控制每次 commit 的边界。所以：**工作区 → 暂存区（挑选）→ 提交（打包）**。

### 3.2 指令与背后结构

**1）查看当前状态**

```bash
git status
```

- **在干什么**：对比「工作区 vs 暂存区 vs 最近一次 commit」，告诉你：哪些文件被改了还没 add、哪些已经 add 还没 commit、有没有新文件/删除。  
- **背后结构**：Git 通过对比三块区域的内容来给出状态；红色一般是「只在工作区有改动」，绿色是「已在暂存区、待 commit」。

**2）把改动放进暂存区（add）**

```bash
git add 某个文件.txt
```

- **在干什么**：把 `某个文件.txt` 的**当前内容**（工作区）复制到暂存区，标记为「参与下一次 commit」。若文件是新建的，也会被加入暂存区。  
- **背后结构**：暂存区里存的是「文件快照」，不是「 diff」；所以再次修改同一文件后，需要再执行一次 `git add` 才能把最新改动纳入暂存区。

```bash
git add .
```

- **在干什么**：把当前目录及子目录下**所有已跟踪文件的改动** + **所有新文件** 都放进暂存区。  
- **注意**：被 `.gitignore` 忽略的文件不会被 add；删除的文件要用 `git add 被删文件` 或 `git add -u` 才会把「删除」这个动作放进暂存区。

**3）把暂存区打成一条提交（commit）**

```bash
git commit -m "这里写本次提交的简短说明"
```

- **在干什么**：  
  1. 把暂存区里所有内容打成一条**快照**（commit）。  
  2. 写入本地仓库（.git），并生成一个唯一的 **commit ID（哈希）**。  
  3. 当前分支（一般是 `main` 或 `master`）会指向这条新提交。  
- **背后结构**：commit 之后，暂存区内容已经「固化」进历史，但工作区文件不变；若你再改文件，又会出现「工作区 vs 暂存区」的差异，需要再次 add → commit。  
- **-m "..."**：直接在命令行写提交说明；不写 -m 会打开编辑器让你写，建议养成每次 commit 都写清楚「做了什么」的习惯。

**4）常见组合：先看再 add 再 commit**

```bash
git status
git add .
git commit -m "添加 GitHub 学习笔记目录与 01-03"
```

- **在干什么**：确认改动 → 全部纳入暂存区 → 打成一条提交。这是最常用的「本地提交」流程。

---

### 3.3 提交说明（commit message）怎么写

- **一句话说清「做了什么」**，例如：「添加 04 add 与 commit 笔记」「修复首页链接」「更新依赖版本」。  
- 若团队有规范，可能要求：第一行简短，空一行再写详细说明；个人项目至少保证第一行有意义，便于以后用 `git log` 回顾。

---

## 4. 总结 / 与其它知识的联系

- **add** = 把工作区的改动「勾选」进暂存区；**commit** = 把暂存区「打包」成一条永久记录，存入本地仓库。  
- 上传到 GitHub 不会自动发生；commit 只影响本地。要传到远程，必须再执行 **git push**（见 05、06）。  
- 日常节奏就是：改文件 → `git add`（选哪些）→ `git commit -m "说明"`（打成一条）→ 需要时 `git push`（同步到远程）。
